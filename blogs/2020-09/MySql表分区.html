Mysql表分区

周鸿宇诶 2020-09-24 MySql

业务场景

​ 在近期项目中的分析模块开发中，由于Redis内存压力，需要将数据转存到Mysql，报表分析模块转Java端开发。该表主要为合约组关联的历史行情信息，大概每天的转存量为200W-300W条左右数据。单纯代码查询和索引优化拉取报表已经没办法满足查询速度的要求（表的体积太大）。 ​ 分析：经过业务分析，合约组的编号大概为60条左右，因此优化方案为：行情波动信息按月分表，对分表根据合约组编号做范围分区，建联合索引查询。 ​ 之前并不是很了解MySql的表分区，因此查漏补缺一下。

# 概念

  • Mysql从5.1版本开始支持表分区，通俗的讲表分区就是将一张大表在物理层面上分割成若干的小表，其实类似于我们电脑中的磁盘分区。

# 表分区的优点

  • 改善大型表以及各种访问模式的的表的可伸缩性，可管理型和提高数据库效率。
  • 与单个磁盘或文件系统分区相比，可以存储更多的数据。（单个磁盘和文件系统对单个文件的大小有限制，通过分区后，我们可以将不同的分区存储到不同的硬盘上）
  • 方便后期维护数据库表：例如，某些历史行情数据已经没有什么价值，我们就可以通过删除分区的方式直接从磁盘上抹去这一部分历史数据，减小表的体积，改善表的性能。delete语句需要一条一条去删除，性能比较低。
  • 对于开发者而言我觉得最大的优点就是合理的运用表分区能极大的提高查询性能
  • 通过跨磁盘甚至跨服务器分散查询数据，能得到更高的查询吞吐量。
  • 在Mysql的5.5版本后支持所有的分区函数化

# 基本分区

  • RANGE分区：基于一个给定连续区间的列值，把多行分配给分区。
  • List分区：类似于RANGE分区，区别是LIST分区是基于列值匹配一个离散集合中的某个值来进行选择。
  • HASH分区:基于用户定义的表达式返回值来进行选择的分区。表达式将要插入到表中的这些行的列值进行计算。这个函数包括MySql中有效的、返回非负整数的任意表达式。
  • KEY分区:类似于HASH分区，区别是KEY分区只支持一列的计算，支持的数据离散分布均匀程度和数据量有所不同。

# RANGE分区

  • 基于一个给定连续区间的列值，把多行分配给分区。
  • 这些区间连续且不能相互重叠，可以使用VALUES、LESS、THAN操作符来定义。
/* 
	更新表分区语句 
	以下语句意思可以翻译成：
	将titles进行范围分区，分区条件为年份的值。
	p01分区代表小于1985年的数据
	p02分区代表大于1985小于1986的数据
	....
	p04分区代表大于1987年以上的数据
*/
ALTER TABLE titles
partition by range(year(from_date))
(
  partition p01 values less than (1985),
  partition p02 values less than (1986),
  partition p03 values less than (1987),
  partition p04 values less than (MAXVALUE)
);

/*
	创建表时候建分区,根据age步长为10进行分区。
	这里要注意分区的名字是不区分大小写的！！！！ 
	同时分区条件一定要包含主键，唯一约束列！！！！
*/
CREATE TABLE T1(
  `id` int(11) NOT NULL,
  `name` varchar(45) NOT NULL,
  `age` int(3) NOT NULL
)partition by range(age)
(
  partition p01 values less than (10),
  partition p02 values less than (20),
  partition p03 values less than (30),
  partition p04 values less than (MAXVALUE)
);

# LIST分区

类似于RANGE分区，区别是LIST分区是基于列值匹配一个离散集合中的某个值来进行选择。

语法：PARTITION BY LIST(expr)

其中“expr”是某一列的值，或者基于某一列的值、并返回一个整数的表达式。然后通过VALUES IN (value_list)来定义每个分区

value_list代表用逗号分割的整数列表，例如(1,2,3,4,5)。

在Mysql5.1版本使用LIST分区，默认只能匹配整数列表

在5.5版本后通过COLUMNS关键字可以支持字符串，日期列作为分区列。

/* 例子 
	 将T2表根据商品类别分区，类别编号为123，456，789分区到p01,p02,p03
	 解释：
	 在上文概念中提高的离散集合中的某个值，也就是说LIST分区给我们提供了更高的灵活性
	 假如我们对T2的类别做范围分区1-2，3-5，6-8。
	 突然产品经理加了个编号为10的商品，需要该商品分区到p01。
	 很显然RANGE这种基于连续区间的分区没法做到，
	 只能借助更为灵活的LIST分区。
*/
create table T2(
 id int,
 cid int comment '商品类别',
 name varchar(20),
 pos_date datetime
)
partition by list(cid)
(
  partition p01 values in (1,2,3),
  partition p01 values in (4,5,6),
  partition p01 values in (7,8,9)
);

# HASH分区

基于用户定义的表达式返回值来进行选择的分区。表达式将要插入到表中的这些行的列值进行计算。这个函数包括MySql中有效的、返回非负整数的任意表达式。

还是用T2商品表创建HASH分区

/* HASH分区语法 */
create table T2(
 id int,
 cid int comment '商品类别',
 name varchar(20),
 pos_date datetime
)
partition by HASH(cid)
partitions 4; /*根据cid分四个区 */

# LINEAR HASH分区

线性HASH分区，同样属于HASH分区的一种。使用的是一种线性的2次幂运算法则，常规HASH使用的是求HASH函数的模数。（可以联系JAVA中的HashMap的知识点理解）。与常规的HASH分区比较：

  1. 优点是增加，删除，合并和拆分分区将变得更快捷。有利于处理极大数量的表。(TB级)

  2. 缺点是数据分散可能不是那么均匀，容易产生”hotspot nodes“热点节点问题。由于分布不均匀可能导致某些分区访问压力过大，某些分区访问压力比较轻。

/* LINEAR HASH分区语法 */
create table T2(
 id int,
 cid int comment '商品类别',
 name varchar(20),
 pos_date datetime
)
partition by linear hash(cid)
partitions 4; /*根据cid分四个区 */

# KEY 分区

使用频率不高

类似于HASH分区，区别是KEY分区只支持一列的计算，支持的数据离散分布均匀程度和数据量有所不同。

与HASH分区的区别是，HASH分区计算可以根据用户定义的表达式来分区。而KEY分区是由MySql服务器提供。本质上都是做了Hash运算。

/* KEY 分区语法 */
create table T2(
 id int,
 cid int comment '商品类别',
 name varchar(20),
 pos_date datetime
)
partition by linear key(cid)
partitions 4; /*根据cid分四个区 */

# 多列分区

在MySql 5.5 版本后支持多列分区，通过COLUMNS关键字允许字符串，日期作为分区定义列，同时还允许使用多个列定义一个分区

/* 语法 */
createe table (
 id int,
 age int,
 weight int
)
partition by range columns(id,age)
(
  /* 这里的多列匹配原则类似于联合索引最左匹配原则，先匹配id匹配到了就不会去匹配age */
	partition p01 values less than (10,34),
  partition p02 values less than (20,54),
  partition p03 values less than (30,18),
  partition p04 values less than (MAXVALUE,MAXVALUE)
);

# 子分区

顾名思义，子分区就是对每个分区再进行分区。

子分区可以用于大表，在多个磁盘间分配数据和索引。

/* 语法 */
CREATE TABLE T3(
	id INT,
  udate DATE
)
PARTITION BY RANGE(YEAR(udate)) -- 对T3表按照udate的年份进行范围分区（暂成为主分区）
SUBPARTITION BY HASH(TO_DAYS(udate)) -- 针对以上的每个主分区再对udate的日进行HASH分区，分区数为2
SUBPARTITIONS 2
(
  PARTITION P01 VALUES LESS THAN (1990),
  /*
  为了优化I/O存储性能，可以将分区对应的文件分散到各个磁盘上进行存储。
  data directory = 'disk_a/...'
  index directory = 'disk_b/...',
  */
  PARTITION P02 VALUES LESS THAN (2000),
  PARTITION P03 VALUES LESS THAN MAXVALUE
);
-- 因此以上分区总共会生成6个分区。

# Range&List分区管理

  • 值得注意的是删除分区同时间会删除分区里的所有数据。如果仅仅只是为了抹去数据，一般建议使用TRUNCATE语句。
alter table ** drop partition p01;
alter table ** truncate partition p02;
1
2
  • 新增分区
alter table ** add partition (partition p* values less than **); --RANGE
alter table ** add partition (partition p* values in (.......)); --LIST
1
2
  1. 对于RANGE新增分区，只能从高端去加。

例：如果某表原来根据年龄分区{(0,20),(21,30),(31,40)}分区

现在想新增分区那么只能从41开始新增分区，破坏了原来分区的定义，因为低端区的数据已经存放到对应的分区文件中。

  1. 同样对于LIST分区为了不破坏原来分区定义同样新增分区不能包含现有分区的列值条件。
  • 分区重组（PEORGANIZE）

使用PEORGANIZE可以对现有分区进行重组，将一个分区拆分成几个分区，或者将几个分区组成一个分区

--拆分区
ALTER TABLE ** PEORGANIZE PARTITION p01 |NTO(
	partition s0 values less than (5),
  partition s1 values less than (10)
);
--分区合并
ALTER TABLE ** PEORGANIZE PARTITION p01 |NTO(
	partition p01 values less than (10)
);

# HASH&KEY分区管理

  • 对于HASH和KEY分区，可以使用COALESCE缩减分区数量，ALTER ... ADD PARTITION增加分区数量
alter table t2 coalesce partition 2;
alter table t2 add partition partitions 2;

# 分区维护

  • 整理分区碎片
    • 抽取分区数据保存临时表，然后删除分区里面数据，然后再插入回去
    • IO消耗很大
ALTER TABLE T1 REBUILD PARTITION P1,P2,P3;
1

# 优化分区（OPTIMIZE）

  • 对于经常进行修改和删除的分区，会产生较多的碎片。
ALTER TABLE T1 OPTIMIZE PARTITION P1,P2,P3;
1
  • 以上语句主要用来回收没有使用的空间，并整理分区数据文件中的分区碎片。其IO消耗相比REBUILD小一点。

# 检查与修复分区

ALTER TABLE T1 CHECK PARTITION P1,P2,P3;
